\section{Streaming and Deployment Evidence}

The most dramatic gains appear when the model keeps revisiting nearly the same
visual state. Deployment-scale evaluations on a larger Gemma 4 26B stack
report 0.8\,s median same-video follow-up latency at 10--18$\times$ speedup,
paired Streaming VideoMME results with 13$\times$ fewer ViT calls and
\mbox{$\sim$50$\times$} less dominant-pipeline compute, and selected
real-video regimes with 4.2--4.5$\times$ end-to-end speedups. A separate
large-$N$ exactness result shows byte-identical Qwen outputs across 1,937
sparse benchmark items when unchanged regions are reused instead of being
re-encoded. Live-camera measurements then show the same regime dependence in
the wild: ViT savings range from 5$\times$ to 300$\times$ with scene activity,
and the post-fix RTSP demo still reduces ViT calls by 14.3$\times$ over an
active 60\,s window. This is the operational regime where the thesis matters
most.

Smaller-$N$ UI studies are still useful as illustrations, but they are not
asked to carry the main deployment claim. The sectional-scroll and RTSP demos
therefore stay explicitly weaker than the paired streaming and real-video
measurements above: they show the same anti-recomputation loop behaving
coherently on active content, but the main-body evidence is the deployment
table below.

The current streaming evidence does not yet close a full systems-reviewer
checklist. The paper still lacks the screenshot-polling baseline, broader UI
diversity, and a dedicated event-detection benchmark. What it already shows is
that the same anti-recomputation principle survives outside sparse benchmark
slices and into the regime where users actually keep asking the same system
about nearly the same scene.
